# branch-commit-push

トランクブランチ（develop/main/master）で作業してしまった未コミットの変更を、新しいブランチに退避し、意味のある単位に分けてコミット、リモートにプッシュする。

## 手順

### Step 1: 現在の状態を確認

```bash
git branch --show-current
git status
git diff
```

### Step 2: ブランチの用意

現在のブランチ名を確認する。

- **develop / main / master にいる場合**: 新しいブランチを切る
- **すでに作業ブランチ（develop/main/master以外）にいる場合**: 新しいブランチを切るかどうかをユーザーに確認する。「このまま今のブランチにコミットしますか？それとも新しいブランチを切りますか？」

トランクブランチにいる場合、そのブランチに対応する派生元候補（例: `develop` とその追跡先 `origin/develop`）が複数存在することがある。それらのコミットハッシュを比較し、**すべて同一であれば派生元の確認は不要**。ハッシュが異なる場合は、どちらを派生元とするかユーザーに確認する。

新しいブランチを切る場合、**毎回**「ブランチ名を指定しますか？それとも自動生成しますか？」とユーザーに確認する。

#### ユーザーが名前を指定する場合

入力された名前をそのまま使う（`type/description` のような形式を強制しない）。チケットID運用のプロジェクトでは、ブランチ名の末尾に `XXX-9999` のようなチケットIDを含めてもらう。

#### 自動生成する場合

1. Step 1で確認した `git diff` の内容から、変更の性質（新機能・バグ修正・リファクタ等）と短い英語の説明を読み取る
2. `type/description`（英語ケバブケース）の形式でブランチ名を組み立てる。typeは `commit` スキルのType一覧（`feat` / `fix` / `refactor` / `docs` / `style` / `perf` / `test` / `chore` / `ci` / `build`）に合わせる
   - 例: 割引率オーバーフローの不具合修正 → `fix/discount-rate-overflow`
4. 組み立てたブランチ名を**必ずユーザーに提示し、確認を取ってから**作成する。気に入らなければ調整に応じる

```bash
git checkout -b <ブランチ名>
```

### Step 3: コミット

Skillツールで `commit` スキルを起動し、その手順（変更内容の確認 → チケットID要否の判定 → 意味のある単位への分割 → ステージングとコミット）に従ってコミットする。コミットメッセージ形式・チケットID対応のルールもすべてそちらに従うこと。

### Step 4: リモートへプッシュ

すべてのコミットが完了したら、まとめてプッシュする。

```bash
git push -u origin <branch-name>
```

## 重要な注意事項

- **秘密情報ファイル（`.env`, `credentials.json` 等）は絶対にステージングしない**
- ブランチ名は、実行前に必ずユーザーに提示して確認を取る
