# git-change-report

現在のブランチで、分岐元ブランチから積んだコミットの内容を、障害チケット用とプルリク用の2種類のMarkdown文章にまとめる。

## 全体の流れ

1. 分岐元ブランチを自動検出する
2. 対象コミットの一覧とdiffを収集する
3. チケット用・PR用のMarkdown文章を作成する
4. ユーザーに提示し、量や粒度の希望を聞いて調整する

## 1. 分岐元ブランチの自動検出

`scripts/detect_base_branch.sh` を実行する。

```bash
bash <skill-path>/scripts/detect_base_branch.sh
```

このスクリプトは、ローカル・リモートブランチのうち **`main`/`master`/`develop`/`release` のいずれかを名前に含むブランチのみ**を候補にして、HEADとのmerge-base（3-way mergeでいう共通祖先＝分岐点）を比較し、**最も新しい分岐点を持つブランチ**を「切った元のブランチ」として出力する。featureブランチ同士などノイズになりうるブランチは候補から除外し、トランクとなりうるブランチだけを見ることで誤検出を減らしている。

- 成功時: `base_branch=develop` `merge_base=<sha>` の形式で出力される。この場合も、生成前に検出結果をユーザーに一度提示し「developから分岐したブランチとして進めます」のように確認を取ってから次に進むこと
- 分岐点が同時期で複数候補がある場合や、該当する候補が見つからない場合はその旨が出力される（exit 1）。この場合は必ずユーザーに「どのブランチから分岐しましたか？」と確認する
- **現在のブランチ自体が `main`/`master`/`develop`/`release` に該当する場合（exit 3）**: 「ブランチを切った地点」という前提が成り立たないため自動検出は行われない。この場合は必ずユーザーに次の2点を確認すること
  1. 本当にこのブランチで処理を進めてよいか（トランクブランチ上で作業している可能性があるため）
  2. 直近何コミットを対象にするか（`git log -N` のNをユーザーに指定してもらう）
  
  確認が取れたら、`git log -<N>` や `git diff HEAD~<N>..HEAD` のように件数ベースで対象範囲を扱う（merge-baseは使わない）。

## 2. 対象コミットの収集

検出した `merge_base` を使って、分岐後のコミットとdiffを取得する。

```bash
git log <merge_base>..HEAD --no-color
git diff <merge_base>..HEAD --stat --no-color
git diff <merge_base>..HEAD --no-color
```

コミットメッセージだけでは「なぜその変更が必要だったか（原因）」が分からないことが多いので、必要に応じて実際のdiffの中身まで読み、変更の意図を把握する。

## 3. Markdown文章の作成

障害チケット用とPR用は同じ変更を扱うが、読み手が異なるため力点を変える。

- **障害チケット**: 読み手は運用・PMなど非エンジニアも含む。「何が起きて、なぜ起きたか」という原因の説明に重心を置く
- **PR**: 読み手はコードをレビューする開発者。「何を実装・修正したか」という変更内容そのものに重心を置く

それぞれ以下のテンプレートに沿って、コピー可能なMarkdownコードブロックとして出力する。

### 障害チケット用テンプレート

```markdown
## 概要
（何が起きたかを一言で）

## 原因
（コミット・diffから読み取れる原因。憶測で断定しない）

## 対応内容
- （変更点を箇条書きで）
```

### PR用テンプレート

```markdown
## 変更内容
- （実装・修正した内容を箇条書きで）
```

原因や変更内容は、必ずコミットメッセージやdiffの内容を根拠にすること。diffだけでは原因が判断できない場合（例: 単なる書き方の癖なのか不具合修正なのか読み取れない場合)は、推測で埋めずにユーザーに確認する。

## 4. 量・粒度の調整

最初は上記テンプレート程度の標準的な粒度で下書きを出力する。ユーザーは変更内容やチケットの重さによって欲しい詳細度が変わるので、出力後に「もっと簡潔に」「もっと詳しく」「この項目は要らない」といった要望を聞き、その場で調整する。一度で完璧な粒度を当てにいく必要はない。
