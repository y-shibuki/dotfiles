# commit

現在の変更を確認し、意味のある単位に分けてコミットする。プッシュは行わない（プッシュも必要な場合は `commit-push` や `branch-commit-push` を使う）。

## 手順

### Step 1: 変更内容の確認

```bash
git status
git diff
git log --oneline -5 --no-color
```

- `git log` は必ず `--no-color` を指定し、ANSIカラーコードを除外する
- 直近のコミットメッセージのスタイル（type、粒度、文体）を確認し、揃える

### Step 2: チケットID要否の判定

現在のブランチ名を `/` で分割し、**最後のセグメントが `英字-数字`（例: `XXX-9999`）の形式にマッチするか**を確認する。

```bash
git branch --show-current
```

- マッチする場合: そのセグメントをチケットIDとして扱い、コミットメッセージの先頭に含める
- マッチしない場合: チケットIDなしの通常形式でよい

判定はブランチ名だけを見て機械的に行い、ユーザーに毎回確認する必要はない。ただし判定結果（チケットIDありと判断したこと）は、コミット前にユーザーに一言伝えること。

### Step 3: 変更を意味のある単位に分割する

1. `git status` と `git diff` で変更されたファイルとその内容を把握する
2. 関連するファイル同士をグルーピングし、複数のコミットに分ける案を作る（例: 機能Aに関するファイル群でひとつ、設定ファイルの変更でもうひとつ、など）
3. **分割案をユーザーに提示し、確認を取ってから実行する**。「以下の単位でN回に分けてコミットします」という形で、各コミットに含めるファイルと簡単な説明を示す

### Step 4: ステージングとコミットの実行

分割案が確定したら、コミット単位ごとに以下を実行する。

```bash
git add <specific-files>
git diff --cached
git commit -m "<commit message>"
```

- **`git add .` や `git add -A` は使用しない**。必ず関連ファイルのみを個別に指定する
- コミット前に `git diff --cached` でステージング内容を確認する
- 自動生成ファイルや一時ファイルが混ざっていないか確認する
- **秘密情報ファイル（`.env`, `credentials.json` 等）は絶対にステージングしない**

## コミットメッセージ形式

Conventional Commitsをベースにする。

```
<type>: <description>
```

チケットIDが必要な場合（Step 2でマッチした場合）は、先頭にチケットIDを付ける。

```
<TICKET-ID>: <type>: <description>
```

**例:**
- チケットIDなし: `feat: ログイン後のリダイレクト先を設定ページに変更する`
- チケットIDあり（ブランチ名が `feature/login-redirect/XXX-9999` の場合）: `XXX-9999: fix: ログイン後のリダイレクト先が正しく設定されない不具合を修正する`

### Type一覧

| type | 用途 |
|------|------|
| `feat` | 新機能 |
| `fix` | バグ修正 |
| `docs` | ドキュメントのみの変更 |
| `style` | コードの意味に影響しない変更（空白、フォーマット等） |
| `refactor` | バグ修正や機能追加を伴わないコード変更 |
| `perf` | パフォーマンス改善 |
| `test` | テストの追加・修正 |
| `chore` | ビルドプロセスやツールの変更 |
| `ci` | CI関連の変更 |
| `build` | ビルドシステムや依存関係の変更 |

### メッセージのガイドライン

- descriptionは日本語で記述する
- 1行で変更内容を簡潔に説明する（50文字以内推奨）
- 現在形・命令形で記述する（「〜を追加する」「〜を修正する」）
- 複数の変更がある場合は、主要な変更に焦点を当てる
